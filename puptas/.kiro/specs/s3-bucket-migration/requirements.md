# Requirements Document

## Introduction

This feature migrates the PUPTAS file storage system from local filesystem storage (`storage/app/public/...`) to an S3-compatible storage backend (Railway Storage Buckets). The migration must preserve all existing upload and retrieval flows, maintain backward compatibility for files already stored locally, and introduce a `FileService` abstraction layer that allows switching between local and S3 storage via environment configuration. Railway buckets are private by default, so all file access must go through the backend using signed or temporary URLs — never direct public bucket URLs.

The system currently uses `ImageCompressionService` to compress and convert uploads to WebP, then stores them via `Storage::disk('public')`. `FileMapper::resolveDiskForPath()` already probes multiple disks to locate a file, providing a foundation for backward compatibility. `config/filesystems.php` already defines an `s3` disk entry using environment variables.

## Glossary

- **FileService**: A new Laravel service class (`App\Services\FileService`) that abstracts all file storage and retrieval operations, delegating to the configured disk.
- **Active_Disk**: The filesystem disk currently in use, resolved from the `FILESYSTEM_DISK` environment variable (values: `public`, `s3`).
- **S3_Disk**: The `s3` disk defined in `config/filesystems.php`, backed by Railway Storage Buckets via the AWS-compatible SDK.
- **Local_Disk**: The `public` disk defined in `config/filesystems.php`, backed by `storage/app/public`.
- **UserFile**: The Eloquent model (`App\Models\UserFile`) that stores file metadata including `file_path` (relative path only, never a full URL).
- **ImageCompressionService**: The existing service that validates, resizes, and converts uploaded images to WebP format before storage.
- **FileMapper**: The existing helper class that maps file types, builds signed preview URLs, and resolves which disk a stored path lives on.
- **Temporary_URL**: A time-limited, pre-signed URL generated by `Storage::disk('s3')->temporaryUrl(...)` that grants read access to a private S3 object.
- **Signed_Preview_URL**: The existing Laravel signed route URL (`files.preview`) used to serve files through the backend controller.
- **Presigned_Upload_URL**: A time-limited URL generated by the S3 SDK that allows a client to upload a file directly to S3 without routing through the backend server.
- **Legacy_Path**: A `file_path` value stored in the database before this migration, pointing to a file on the Local_Disk.
- **Migration_Path**: A `file_path` value stored after this migration, pointing to a file on the S3_Disk.

## Requirements

### Requirement 1: S3 Disk Configuration

**User Story:** As a system administrator, I want the S3 disk to be correctly configured for Railway Storage Buckets, so that the application can connect to the private bucket using environment-provided credentials without hardcoded values.

#### Acceptance Criteria

1. THE S3_Disk SHALL be configured in `config/filesystems.php` using only environment variables: `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_DEFAULT_REGION`, `AWS_BUCKET`, `AWS_ENDPOINT`, and `AWS_USE_PATH_STYLE_ENDPOINT`.
2. THE S3_Disk SHALL set `use_path_style_endpoint` to `true` by default (via `AWS_USE_PATH_STYLE_ENDPOINT=true`) to ensure compatibility with Railway's custom endpoint at `https://storage.railway.app`.
3. THE S3_Disk SHALL NOT expose a public `url` value that assumes public bucket access; the `url` key SHALL remain unset or point to the backend preview route.
4. IF the `AWS_ENDPOINT` environment variable is absent at runtime, THEN THE S3_Disk SHALL throw a configuration exception and THE System SHALL log the missing variable name before aborting the request.
5. THE System SHALL support switching the Active_Disk between `public` and `s3` exclusively via the `FILESYSTEM_DISK` environment variable without requiring code changes.

---

### Requirement 2: FileService Abstraction Layer

**User Story:** As a developer, I want a single `FileService` class to handle all file storage and deletion operations, so that storage backend changes require modifications in only one place.

#### Acceptance Criteria

1. THE FileService SHALL expose a `store(UploadedFile $file, string $directory): array` method that delegates compression to `ImageCompressionService` and persists the result to the Active_Disk, returning `['path' => string, 'original_name' => string]`.
2. THE FileService SHALL expose a `delete(string $path): void` method that removes the file from whichever disk the path resolves to, using `FileMapper::resolveDiskForPath($path)`.
3. THE FileService SHALL expose a `url(UserFile $file): string` method that returns a Signed_Preview_URL for both local and S3 files, so the frontend always receives the same URL shape regardless of the Active_Disk.
4. WHEN the Active_Disk is `s3`, THE FileService SHALL store files under the path prefix `uploads/files/` within the S3 bucket, preserving the same relative path structure used by the Local_Disk.
5. WHEN the Active_Disk is `public`, THE FileService SHALL behave identically to the current `ImageCompressionService` + `Storage::disk('public')` flow, producing no behavioral change for local deployments.
6. IF a storage operation fails due to an S3 connectivity error, THEN THE FileService SHALL throw a `\RuntimeException` with a descriptive internal message, and THE calling controller SHALL catch it, log the full exception, and return a generic 500 response to the client.

---

### Requirement 3: Replace Direct Disk Calls with FileService

**User Story:** As a developer, I want all upload and deletion calls to go through `FileService`, so that the codebase has a single, consistent storage interface.

#### Acceptance Criteria

1. THE `ImageCompressionService::compress()` method SHALL continue to handle image processing (resize, WebP conversion) but SHALL NOT call `Storage::disk('public')->put()` directly; instead, THE FileService SHALL call `ImageCompressionService` and then persist the result.
2. THE `ConfirmationService::reuploadFile()` method SHALL delegate file storage to `FileService::store()` and file deletion to `FileService::delete()` instead of calling `ImageCompressionService` and `Storage::disk('public')` directly.
3. THE `UserFileController::uploadFiles()` method SHALL delegate file storage to `FileService::store()` instead of calling `ImageCompressionService::compress()` directly.
4. THE `ConfirmationService::deleteExistingFile()` private method SHALL delegate to `FileService::delete()` instead of calling `Storage::disk('public')->delete()` directly.
5. WHEN a file is stored via FileService, THE System SHALL save only the relative `file_path` (e.g., `uploads/files/filename.webp`) in the `UserFile.file_path` column — never a full URL or bucket-qualified path.

---

### Requirement 4: Backward-Compatible File Retrieval

**User Story:** As an applicant or staff member, I want to access files uploaded before the migration, so that no previously submitted documents become inaccessible after the migration is deployed.

#### Acceptance Criteria

1. WHEN a file preview is requested via `UserFileController::preview()`, THE System SHALL resolve the correct disk using `FileMapper::resolveDiskForPath($file->file_path)` before attempting to read the file.
2. WHEN `FileMapper::resolveDiskForPath()` is called with a Legacy_Path, THE FileMapper SHALL return `public` if the file exists on the Local_Disk, even when the Active_Disk is `s3`.
3. WHEN `FileMapper::resolveDiskForPath()` is called with a Migration_Path, THE FileMapper SHALL return `s3` if the file exists on the S3_Disk.
4. IF a file path does not exist on any probed disk, THEN THE `UserFileController::preview()` method SHALL return HTTP 404.
5. THE `FileMapper::resolveDiskForPath()` method SHALL probe disks in the order: Active_Disk first, then `public`, then `local`, then `s3`, stopping at the first disk where the file exists.

---

### Requirement 5: Private S3 File Access via Backend

**User Story:** As a system architect, I want all S3 file reads to go through the backend, so that private bucket objects are never exposed via direct public URLs.

#### Acceptance Criteria

1. THE `UserFileController::preview()` method SHALL stream S3 file contents through the backend response when the resolved disk is `s3`, using `Storage::disk('s3')->get($path)`.
2. THE System SHALL NOT generate or expose direct S3 object URLs (e.g., `https://<bucket>.storage.railway.app/...`) to the frontend at any point.
3. THE `FileMapper::buildPreviewUrl()` method SHALL continue to generate Signed_Preview_URLs routed through `UserFileController::preview()` for all files regardless of which disk they reside on.
4. WHEN a Signed_Preview_URL expires (after 60 minutes), THE System SHALL return HTTP 403 to the client.
5. WHERE the `FILESYSTEM_DISK` is set to `s3`, THE FileService SHALL NOT call `Storage::disk('s3')->temporaryUrl()` for serving files to the frontend; all access SHALL go through the Signed_Preview_URL route to maintain a uniform access pattern.

---

### Requirement 6: Presigned Upload URL Endpoint (Optional Advanced Flow)

**User Story:** As a developer, I want a backend endpoint that generates presigned upload URLs, so that large files can be uploaded directly from the Vue frontend to S3 without routing through the application server.

#### Acceptance Criteria

1. WHERE `FILESYSTEM_DISK` is `s3`, THE System SHALL expose a `POST /upload-presigned` route that accepts `{ field: string, content_type: string }` and returns `{ upload_url: string, path: string, expires_in: int }`.
2. WHEN a presigned upload URL is generated, THE System SHALL set the URL expiry to 300 seconds and restrict the allowed content type to the value provided in the request.
3. WHEN the Vue frontend completes a direct S3 upload using the presigned URL, THE System SHALL accept a subsequent `POST /upload-confirm` request containing `{ field: string, path: string, original_name: string }` to create or update the corresponding `UserFile` record.
4. IF the `content_type` in the presigned URL request is not one of `image/jpeg`, `image/png`, `image/webp`, `image/gif`, or `application/pdf`, THEN THE System SHALL return HTTP 422 with a descriptive validation error.
5. THE presigned upload path SHALL follow the pattern `uploads/files/{slug}_{timestamp}_{random}.{ext}` to match the naming convention used by `ImageCompressionService`.

---

### Requirement 7: Error Handling and Logging

**User Story:** As a developer, I want all storage operations to be wrapped in structured error handling, so that failures are logged with enough detail to diagnose issues without exposing internals to end users.

#### Acceptance Criteria

1. WHEN any `FileService` method throws an exception, THE calling controller SHALL catch the exception, log it using `\Log::error()` with the context keys `user_id`, `file_type`, `disk`, and `exception_message`, and return a generic HTTP 500 JSON response with the message `'File operation failed. Please try again.'`.
2. WHEN `Storage::disk('s3')->put()` fails, THE FileService SHALL throw a `\RuntimeException` containing the S3 error code and the attempted file path.
3. WHEN `Storage::disk('s3')->delete()` fails, THE FileService SHALL log a warning using `\Log::warning()` with the file path and error message, and SHALL NOT throw an exception (deletion failures are non-fatal).
4. THE System SHALL NOT include S3 error messages, bucket names, or internal paths in any HTTP response body returned to the client.
5. WHEN an upload request is received and the S3 disk is unreachable, THE System SHALL return HTTP 503 with the message `'Storage service temporarily unavailable.'` after logging the connectivity error.

---

### Requirement 8: File Path Integrity

**User Story:** As a database administrator, I want file paths stored in the database to remain relative and disk-agnostic, so that the database does not need to be updated when switching storage backends.

#### Acceptance Criteria

1. THE System SHALL store only relative file paths in `UserFile.file_path` (e.g., `uploads/files/photo_1234567890_abcd1234.webp`), never absolute paths or full URLs.
2. THE System SHALL NOT prepend bucket names, disk identifiers, or protocol schemes to stored file paths.
3. WHEN the Active_Disk changes from `public` to `s3` (or vice versa), existing `UserFile.file_path` values SHALL remain valid and resolvable via `FileMapper::resolveDiskForPath()` without a database migration.
4. THE FileService SHALL validate that the path returned by `ImageCompressionService` does not contain a leading slash or protocol prefix before persisting it to the database; IF it does, THEN THE FileService SHALL strip the prefix before saving.
